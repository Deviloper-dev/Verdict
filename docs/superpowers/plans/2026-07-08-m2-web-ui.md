# Verdict M2 — Web UI Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:executing-plans. Same-session author/executor format (interfaces + intent; code detail lives in commits).

**Goal:** Member-facing Next.js app: auth, groups, poll creation/voting, record browsing, and the client-side Verify page — dark theme with pastel blue/violet/orange/green accents (PRD §10.1).

**Architecture:** Next.js 15 App Router added to the existing package. Server components + **server actions** call the M1 domain services directly over a `pg` Pool (no separate API layer). Supabase is used for **Auth only** at this layer (`@supabase/ssr`); `members.id = auth user id`, member row upserted on first sign-in. Votes/opinions are hidden while a poll is open (only a voted-count progress is shown) and fully revealed in the sealed record — avoids anchoring bias mid-debate. The Verify page fetches raw records via a route handler and recomputes hashes **in the browser** with the isomorphic chain lib.

**Tech Stack adds:** `next`, `react`, `react-dom`, `@supabase/ssr`, `@supabase/supabase-js`. Plain CSS custom properties for the theme (no UI framework).

## Global Constraints
- Dark theme default; pastel accents: violet/blue = primary/identity, green = finalized/verified, orange = open/awaiting votes.
- All reads scoped by member id in SQL (server actions verify membership); RLS remains defense-in-depth.
- App must `next build` cleanly and unit/DB tests stay green; full auth flow requires a Supabase project (env vars in `.env.example`).
- v1 invite limitation: members can be added to a group by email only after they've signed in once (member row exists). True email invites are a follow-up.
